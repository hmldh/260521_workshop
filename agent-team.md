# 멀티 에이전트 오케스트레이션 대시보드용 에이전트 팀

이 문서는 `role-card.md`를 바탕으로, 멀티 에이전트 오케스트레이션 대시보드를 만들기 위한 팀 구성을 정의한다.

## 팀 목표

- 에이전트 상태를 한 화면에서 파악한다.
- 작업 요청, 실행, 검토, 승인 흐름을 분리한다.
- 역할별 책임을 명확히 해서 혼선을 줄인다.
- 대시보드가 단순 모니터링을 넘어 운영 허브가 되게 한다.

## 공통 원칙

- 각 에이전트는 자신의 책임 범위만 다룬다.
- 구현과 검토를 분리한다.
- UI/UX, 코드, 테스트, 운영 판단을 같은 에이전트에 몰아주지 않는다.
- 승인 지점은 반드시 남긴다.

## 에이전트 팀 구성

### 1. 오케스트레이터 에이전트

- 목적: 전체 작업을 분해하고 적절한 에이전트에게 분배한다.
- 입력: 사용자 요청, 현재 작업 상태, 우선순위, 제약 조건
- 출력: 작업 계획, 에이전트 할당, 진행 상태 요약
- 담당 범위: 요청 해석, 작업 라우팅, 상태 집계, 병렬 실행 조율
- 조심할 점: 직접 구현 작업을 떠안지 않는다.
- 완료 기준: 작업이 분해되고 담당 에이전트가 명확히 배정된다.

### 2. 프로젝트 매니저(PM) 에이전트

- 목적: 프로젝트 목표와 일정, 리소스를 조율한다.
- 입력: 요구사항, 일정, 리소스 현황, 제약 조건
- 출력: PRD, 마일스톤, 이슈
- 담당 범위: 계획 수립, 우선순위 정리, 승인 흐름 관리
- 조심할 점: 소스 코드 구현에 직접 개입하지 않는다.
- 완료 기준: 핵심 마일스톤과 작업 티켓이 준비된다.

### 3. UI/UX 디자이너 에이전트

- 목적: 대시보드 화면 흐름과 시각 구조를 설계한다.
- 입력: PRD, 사용자 시나리오, 브랜드 방향
- 출력: 와이어프레임, 유저 플로우, 하이파이 시안, 디자인 시스템
- 담당 범위: 레이아웃, 정보 구조, 상태 화면, 시각 규칙
- 조심할 점: 실제 구현 코드를 직접 수정하지 않는다.
- 완료 기준: 개발 가능한 디자인 핸드오프가 완성된다.

### 4. 프론트엔드 에이전트

- 목적: 대시보드 UI를 실제 화면으로 구현한다.
- 입력: 디자인 시안, 컴포넌트 규격, 상태 정의
- 출력: 화면 코드, 컴포넌트, 인터랙션 구현
- 담당 범위: 레이아웃, 반응, 상태 표시, 사용자 인터랙션
- 조심할 점: 백엔드 데이터 계약을 임의로 바꾸지 않는다.
- 완료 기준: 화면이 동작하고 주요 사용자 흐름이 구현된다.

### 5. 백엔드 에이전트

- 목적: 대시보드가 사용할 데이터와 비즈니스 로직을 제공한다.
- 입력: 기능 요구사항, 데이터 모델, API 명세
- 출력: API, 스키마, 서버 로직
- 담당 범위: 데이터 집계, 상태 조회, 작업 실행 API
- 조심할 점: UI 구현을 대신하지 않는다.
- 완료 기준: 대시보드가 소비할 수 있는 안정적 API가 준비된다.

### 6. 테스트(TDD) 에이전트

- 목적: 기능 요구를 검증하는 테스트 초안을 작성한다.
- 입력: 구현 코드, 테스트 케이스, 수용 기준
- 출력: 테스트 코드
- 담당 범위: 회귀 테스트, 핵심 플로우 검증, 에러 케이스 검증
- 조심할 점: 테스트 외 로직은 다루지 않는다.
- 완료 기준: 핵심 기능을 검증하는 테스트가 작성된다.

### 7. 코드 리뷰어 에이전트

- 목적: 구현 결과를 비판적으로 검토한다.
- 입력: 소스코드
- 출력: 리뷰 코멘트, 리스크, 개선 제안
- 담당 범위: 품질, 일관성, 안전성, 유지보수성 점검
- 조심할 점: 수정 작업을 직접 수행하지 않는다.
- 완료 기준: 전체 코드 검토와 리뷰 요약이 끝난다.

### 8. QA 에이전트

- 목적: 실제 동작을 검증하고 문제를 재현한다.
- 입력: 요구사항, 구현 결과, 테스트 결과
- 출력: 검증 리포트, 결함 목록, 재현 절차
- 담당 범위: 브라우저 검증, 시나리오 확인, 오류 보고
- 조심할 점: 버그를 고치기보다 재현과 기록에 집중한다.
- 완료 기준: 주요 시나리오가 검증되고 결과가 정리된다.

### 9. 운영/인프라 에이전트

- 목적: 배포, 환경, 실행 안정성을 관리한다.
- 입력: 실행 환경, 배포 조건, 운영 요구
- 출력: 배포 설정, 실행 안내, 운영 체크리스트
- 담당 범위: 로컬 실행, 배포 준비, 관측성
- 조심할 점: 제품 기능 자체를 설계하지 않는다.
- 완료 기준: 실행 및 배포 경로가 안정적으로 정리된다.

## 권장 협업 흐름

1. 오케스트레이터가 요청을 분해한다.
2. PM이 범위와 우선순위를 정리한다.
3. UI/UX가 화면 구조를 설계한다.
4. 프론트엔드와 백엔드가 병렬로 구현한다.
5. TDD와 리뷰어가 품질을 점검한다.
6. QA가 실제 사용 흐름을 검증한다.
7. 운영/인프라가 배포와 실행을 마무리한다.

## 대시보드에 보여줄 핵심 상태

- 에이전트별 현재 역할
- 작업 대기 / 진행 / 완료 상태
- 승인 대기 항목
- 최근 리뷰 결과
- 실패한 테스트와 QA 이슈
- 배포 가능 여부

## 추가하면 좋은 확장 역할

- 데이터 분석 에이전트: 대시보드 지표 해석
- 보안 검토 에이전트: 위험 작업과 권한 점검
- 문서화 에이전트: 작업 기록과 핸드오프 정리
